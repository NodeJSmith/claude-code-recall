## 3. Coverage Gaps

### Gap 1 — AC#4 and AC#5 have no verifying test anywhere in the plan (blocking)

- **AC#4** (FR#6): "After syncing and embedding, `ccrecall search-messages --query "retry task"` ... returns a result with the matching exchange."
- **AC#5** (FR#7): "The search-messages result for a tool-content match includes the tool content in its snippet text."

- T03 claims both in its frontmatter `implements` list, but T03's Target Files only add `tests/test_summarizer.py` (exchange-pair level, not a search-messages call). T03's own Verify checklist item for AC#4 self-admits the gap: *"AC#4: ... (end-to-end, may need integration test in T05)"* — a hedge, not a plan.
- T05 is the obvious place this test would land (depends_on T01-T04, is the integration-test task), but T05's Prompt enumerates exactly 4 tests: tool-only turn persistence, FTS `__tools__` section, backfill populate, backfill skip-missing-file. None of them calls `search_conversations.search_messages` / the CLI `search-messages` command or asserts on a result snippet. T05's frontmatter `implements` is `["AC#10", "AC#11"]` only — it doesn't even claim AC#4/AC#5, so there's no accountability gap to notice by re-reading T05 alone.

**Impact:** the design's own motivating scenario — "Find tool content via search" (`ccrecall search-messages --query "retry task"`) — is never exercised end-to-end by any task in this plan. Nothing will fail CI if this path is broken.

### Gap 2 — `parsing.py`'s modified functions have zero unit-test coverage, before or after

T03 modifies `aggregate_branch_content` and `build_aggregated_content` (adds a `__tools__` section) but its Target Files list only `tests/test_summarizer.py`, not `tests/test_parsing.py`. Confirmed against the repo: `tests/test_parsing.py` exists (57 tests) but currently has zero tests referencing `aggregate_branch_content` or `build_aggregated_content` — so this isn't a regression the plan introduces, but it does mean AC#3's stated criterion ("`aggregated_content` contains tool markers") is unverifiable from T03's own diff; it's deferred entirely to T05's untracked integration test (see Gap/Contradiction below). Since T03 is adding new, non-trivial branching logic (a new section, conditional on non-empty tool_content) to a function with no existing unit coverage, this is worth a dedicated unit test rather than relying solely on one integration assertion.

- **AC#3** (T03's frontmatter claim) is verified by T05's test 2, "FTS includes tool content": *"query `branches.aggregated_content` and assert it contains a `__tools__` section..."* T05's frontmatter does not list AC#3.

This is a traceability-metadata defect: the matrix built from `implements:` fields is not a reliable map of what verifies what for 3 of 11 ACs, even though it happens to be functionally covered (for AC#2/AC#3/AC#6, unlike AC#4/AC#5 which are not covered at all).

### Contradiction 2 — `build_message_row` is described as returning "a dict"; it returns a tuple

Verified against `src/ccrecall/message_ops.py:40-73` (current code):

```python
def build_message_row(...) -> tuple | None:
    ...
    return (
        session_id,
        uuid,
        entry.get("parentUuid"),
        entry.get("timestamp"),
        entry_type,
        text,
        has_thinking,
        int(is_notification),
        parse_origin(entry),
    )
```

It is a positional tuple, consumed positionally by `insert_new_messages` (e.g. `existing_uuids.add(row[1])` for uuid). But:

- **T02 Focus:** *"`message_ops.py:build_message_row` returns a dict — the new `tool_content` key must be added to that dict AND to the INSERT column/value lists in `insert_new_messages`."* — factually wrong; there is no dict or key to add, only a tuple position.
- **T04 Prompt:** *"Call `build_message_row` (from `message_ops`) to get the row dict including the new `tool_content`."* — same error, repeated.
- design.md's own Architecture → Storage layer section compounds this by attributing the INSERT itself to `build_message_row` ("`build_message_row` ... includes it in the INSERT"), when the literal `INSERT INTO messages` statement is in `insert_new_messages` (confirmed at message_ops.py:96). T02's own Prompt step 3 correctly locates the INSERT in `insert_new_messages` (line ~94) — so T02 internally contradicts its own Focus section on this point.

**Impact:** an implementer following the "dict" framing literally would either misread the diff needed (must append a new positional element to the tuple, in the same order the INSERT's column list / `?` placeholders expect) or introduce an unplanned refactor from tuple to dict — not called for anywhere else in the design, and inconsistent with `fetch_branch_messages`'s existing dict-based pattern that T02 correctly describes elsewhere in the same task. For the backfill task (T04), this matters more: reusing `build_message_row`'s tuple output to determine "does this row already exist, and what's its `tool_content` value" requires correct positional indexing, which T04's Prompt doesn't address given it's operating under the wrong "dict" mental model.
