## Additional Findings (not covered by the PASS/FAIL checklist, but verified against the actual repository source)

These were found by cross-checking task-file claims against the current state of `src/ccrecall/` (not just against design.md), since several tasks cite specific line numbers and schema details that are checkable.

**1. [Warning] T02's Focus section contains a fabricated `branches` column list.**
T02's Focus section states: *"The branches table after v1 has columns: id, session_id, leaf_uuid, fork_point_uuid, is_active, started_at, ended_at, **duration_seconds**, exchange_count, **total_cost, total_input_tokens, total_output_tokens, total_cache_read, total_cache_write**, aggregated_content, **summary**, embedding_version, summary_version_at_embed"*.

The actual `branches_new` DDL in `src/ccrecall/db.py:279-298` is: `id, session_id, leaf_uuid, fork_point_uuid, is_active, started_at, ended_at, exchange_count, files_modified, commits, tool_counts, aggregated_content, context_summary, context_summary_json, summary_version, embedding_version, embedding_model, summary_version_at_embed`. Columns named `duration_seconds`, `total_cost`, `total_input_tokens`, `total_output_tokens`, `total_cache_read`, `total_cache_write`, and `summary` do not exist anywhere in the `branches` table — they appear to be copied from an unrelated schema (cost/token tracking) rather than derived from the real DDL. The column *count* (18 total, 17 after dropping `fork_point_uuid`) is coincidentally correct, and this same Focus paragraph also correctly matches `tests/test_db.py`'s `EXPECTED_COLUMNS["branches"]` fixture for the cid-renumbering instructions (verified — the "14 rows shift from cid 4-17 to cid 3-16" claim is accurate). The Prompt section separately (and correctly) tells the implementer to "Reference the v1 rebuild at `db.py:283-315` for the exact column order," which would catch this if followed — but an implementer who copies the Focus section's explicit list verbatim into the `CREATE TABLE branches_new` statement would produce a migration with wrong column names. Not a contradiction with design.md (design.md itself never lists explicit column names), but a factual defect worth correcting in T02 before execution.

**2. [Minor] T04 has ~4-line drift in the boundary between two SQL constants.**
T04 claims `_CANDIDATE_QUERY` spans lines 241-255 and `_SESSION_BY_UUID_QUERY` spans lines 258-265. Actual source has `_CANDIDATE_QUERY` at 241-252 and `_SESSION_BY_UUID_QUERY` at 254-265. Every other function boundary cited in T04 (12 functions checked) was verified exact against `src/ccrecall/hooks/memory_context.py`. Low severity — self-correcting since the Prompt instructs "Read `memory_context.py` fully" before extracting.

**3. [Observation] AC#2/AC#3 have no verify item in T01 or T02.**
Design.md scopes AC#2 (`pytest` passes) and AC#3 (`prek` passes) across FR#1-FR#7 — which includes FR#4/FR#5 (T02) and FR#6/FR#7 (T01) — but only T03/T04/T05 carry AC#2/AC#3 verify checkboxes. This is not a coverage gap (both ACs have a verified home), but it means T01 and T02 in isolation have no explicit full-suite/lint gate of their own; the global pass is only checked once the split tasks land.

**4. [Observation] AC#1's 400-line ceiling doesn't cover files T01/T02 touch.**
AC#1 is explicitly scoped in design.md to FR#1/FR#2/FR#3 only. `session_tail.py` (384 lines, modified by T01) and `db.py` (497 lines already, modified by T02) sit outside AC#1's stated scope. `db.py` already exceeds the project's 400-line "typical" convention ceiling before T02 adds `_migrate_to_v2`; this is a design.md scoping choice rather than a task-file defect, but is worth the implementer's awareness since no task gates it.
