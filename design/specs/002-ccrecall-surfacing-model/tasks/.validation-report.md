## Warnings

### Warning #1 — FR#16 Verify item in T03 covers only the hook-path half

T03 claims FR#16 and its Verify item reads: "a forced exception in the proactive path leaves the normal context injection intact and never breaks the hook." This tests only the hook / proactive-alert half of FR#16.

FR#16 also requires that the reactive-caveat logic ("command" path) never raises. This half is tested in T04 under AC#9 ("forced failure in caveat computation leaves recall results intact"), but T04 does not carry FR#16 in its implements list and has no FR#16 Verify item.

The coverage exists by transitivity (FR#16 → AC#9 in the design; AC#9 is verified in T04), but a traceability auditor looking only at the implements lists will not see that T04 contributes to FR#16. **No functional gap** — but if the plan is re-reviewed, consider adding FR#16 to T04's implements list with a Verify item covering the caveat/recall path.

### Warning #2 — AC#5 end-to-end behavior not explicitly verified in any single task

AC#5 states: "After the underlying condition is fixed, the next session injects no proactive block and the snooze record for that key is reset." This has two halves:

- Snooze-record reset when condition clears: tested by T01 (FR#9 Verify).
- No block injected at the following SessionStart: implicitly covered by T03's restructuring, but T03 has no AC#5 Verify item, and AC#5 is not in T03's implements list.

T02's FR#5 Verify proves the embedding-status sidecar is cleared on success. T01's FR#9 Verify proves the snooze record is dropped. But the combined end-to-end scenario ("embedding status cleared → snooze dropped → next session produces no block") is not a named test case in any task. **No functional gap** — each prerequisite is individually verified — but the integration path for the embedding-failure variant of AC#5 is untested in T03. Consider adding an explicit test case in T03 for "embedding process clears status → next session no block."
