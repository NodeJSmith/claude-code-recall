### T01/T02 finding (blocking)

T01's Target Files list modifies `src/ccrecall/cli/commands.py` and `tests/test_db.py`. T02's Target Files list also modifies both of those same files. Neither task's `depends_on` references the other (`T01: depends_on: []`, `T02: depends_on: []`), so nothing in the task metadata forces sequential execution. If an orchestrator schedules independent tasks in parallel (e.g., each in its own worktree, per this project's own convention for isolating parallel executors), both agents would edit the same two files concurrently — on `cli/commands.py` that's two independent command removals (`cmd_tokens` in T01 vs. `cmd_migrate`/`cmd_write_config` in T02) landing in the same command-registration file; on `tests/test_db.py` it's two independent import/class removals in the same file. This is the "parallel work on shared files is a merge conflict, not parallelism" failure mode.

Notably, **T02's own Focus section is aware of the risk but doesn't resolve it structurally**: *"In `cli/commands.py`, both `cmd_write_config` and `cmd_migrate` are being removed. T01 also removes `cmd_tokens` from this file. If T01 hasn't run yet, only remove the onboarding and legacy commands/imports."* This describes the coordination hazard in prose rather than encoding it as `depends_on: ["T01"]` — it tells the executor what to do *if* T01 hasn't run, but doesn't force ordering, and offers no equivalent guidance at all for the `tests/test_db.py` overlap.

**Recommended fix**: add `depends_on: ["T01"]` to T02 (or the reverse). Cheap fix — T03 already waits on both, so adding the edge doesn't change the critical path length.
